# Benchmark Baseline: 2026-06-01

Status: Internal engineering baseline. Not publishable benchmark evidence.

## Method

Mortar uses JMH 1.37 for JVM benchmarks.

Baseline profile:

```bash
gradlew.bat :java:benchmarks:jmhBaseline -PjmhIncludes=PostgresRenderingBenchmark
gradlew.bat :java:benchmarks:jmhBaseline -PjmhIncludes=ReferenceRenderingBenchmark
gradlew.bat :java:benchmarks:jmhBaseline -PjmhIncludes=JdbcExecutionBenchmark
gradlew.bat :java:benchmarks:jmhBaselineAllocation -PjmhIncludes=JdbcExecutionBenchmark
```

Profile settings:

- 5 warmup iterations;
- 10 measurement iterations;
- 3 forks;
- 1 second per warmup/measurement iteration;
- throughput mode;
- GC profiler for allocation run.

Raw JSON is generated under `java/benchmarks/build/reports/jmh/` and is not
committed because build outputs are machine-local.

## Results

### SQL Rendering

| Benchmark | Throughput |
| --- | ---: |
| Mortar PostgreSQL render | 348,438.640 ops/s ± 9,354.001 |
| jOOQ reference render | 264,167.573 ops/s ± 11,643.942 |
| QueryDSL reference render | 135,810.306 ops/s ± 24,599.811 |

Interpretation:

- Mortar is faster than the current jOOQ and QueryDSL reference rendering
  scenarios in this harness.
- The comparison is scoped to equivalent select rendering shape, not the full
  feature surface of those libraries.

### JDBC Adapter

The JDBC benchmark uses controlled JDBC doubles. It measures adapter overhead,
not database or network throughput.

| Benchmark | Throughput |
| --- | ---: |
| Mortar JDBC fetch | 12,231,256.514 ops/s ± 1,160,790.995 |
| Plain JDBC fetch | 6,898,695.924 ops/s ± 319,056.695 |

Allocation profile:

| Benchmark | Allocation |
| --- | ---: |
| Mortar JDBC fetch | 360.001 B/op ± 0.001 |
| Plain JDBC fetch | 336.001 B/op ± 0.001 |

Interpretation:

- Earlier benchmark shape was unfair because plain JDBC consumed fields while
  Mortar materialized a row list. The benchmark now makes both paths materialize
  the same `ClientRow` list.
- After removing noop logging event materialization from Mortar's default hot
  path, Mortar JDBC overhead is small in this controlled JDBC harness.
- The remaining allocation delta is about 24 B/op and should be tracked, but it
  is not currently a release blocker.

## Fixes From This Review

- `MortarJdbcClient` no longer builds redacted log events when the default noop
  logger is installed.
- `JdbcExecutionBenchmark.plainJdbcFetch` now materializes the same logical row
  shape as `mortarJdbcFetch`.
- Gradle now exposes reproducible `jmhBaseline` and `jmhBaselineAllocation`
  tasks with JSON output.

## Limitations

- Controlled JDBC-double results do not predict real PostgreSQL latency.
- Rendering benchmarks do not measure database planning or execution.
- Results should be compared only on the same machine, JDK, command profile, and
  benchmark source.
- Public claims need a release commit, hardware/OS/JDK metadata, raw JSON
  artifacts, and reviewer sign-off.
